The following example involves a situation where I want to analyze historical data on Bay Area Rapid Transit (BART) ridership at the station/hour level. BART kindly makes such ridership information publicly available on their open data portal. This post will examine the following workflow:

  1. Use bs4 and requests to reverse engineer an API for the data and batch download it.
  2. Use the apache parquet to efficiently store the data locally.
  3. Use dask for highly efficient read-in of the data.

I should note that my adoption of dask and parquet storage is due to the influence of two of my day-job coworkers, B. Hasan and Konrad Franco, after we leveraged them for an internal project with a similar workflow.

Data Retrieval

To start, I’ll set up a scraping script using requests and BeautifulSoup to build access to the BART hourly ridership data.

# URL of the webpage to scrape
url = 'https://afcweb.bart.gov/ridership/origin-destination/'
# Send an HTTP GET request to the webpage
response = requests.get(url)
# Check if the request was successful
if response.status_code == 200:
    # Parse the content of the webpage with Beautiful Soup
    soup = BeautifulSoup(response.text, 'html.parser')
else:
    print("Failed to retrieve the webpage")

We’ll proceed to download all of this into a folder, data. Here I take advantage of tqdm’s progress bar so that I can track the potentially large job’s progress. I also add logic for two conditions:

  • make sure that the ingest doesn’t re-read files that I already store locally.
  • UNLESS, it the data is from the current year, in which case it likely contains more data than the present file for that year.
current_year: str = str(datetime.today().year)
# Create the "data" folder if it doesn't exist
if not os.path.exists('data'):
    os.makedirs('data')
# Download and save the files
for url in tqdm(file_urls):
    filename = os.path.join('data', os.path.basename(url))
    current_year_data: bool = re.search("\d{4}", url)[0] == current_year
    file_exists: bool = os.path.exists(filename)
    if file_exists and not current_year_data:
        pass
    else:
        response = requests.get(url)
        if response.status_code == 200:
            with open(filename, 'wb') as file:
                file.write(response.content)
        else:
            print(f"Failed to download: {url}")

Efficient Storage

Since these are in a usable form now, I decided against implementing any other processing steps to the data and preserve a raw form in storage, but I’ll proceed to batch convert this directory and set of files into a directory of parquet files for more efficient data I/O.

To do that, I’ll lazily read in the full directory of .csv files using dask:

data = dd.read_csv(data_paths, blocksize=None, compression='gzip')

Using the Data

With the data in parquet, it’s now very easy to read in and analyze all of this ridership data. For example, here I use dask to read in about 52 million rows of data in under 30 seconds, in a single line of code.

df = dd.read_parquet('data/parquet_data').compute()
[########################################] | 100% Completed | 13.31 s
df.shape
(52063941, 5)

This data is now all in memory as a pandas dataframe, ready for typical use.

type(df)
pandas.core.frame.DataFrame

We can compare that performance to the case where instead of converting to parquet and using dask for data i/o, we simply use the .csvs with pandas for read-in:

dfs = []
start = time.time()
for csv_path in tqdm(data_paths):
    dfs.append(pd.read_csv(csv_path))
df_pandas = pd.concat(dfs)
end = time.time()
print("Seconds elapsed:", end-start)
100%|██████████| 7/7 [00:38<00:00,  5.53s/it]
Seconds elapsed: 76.59228587150574

We can see via tqdm that reading the .csv files took a little over 30 seconds, but the full process – reading in the .csv files and concatenating all of them to produce output equivalent to the dask+parquet read-in– took a little over a minute.

Some reflection

This difference – ~70 seconds versus ~15 seconds – isn’t so massive in the context of this 52 million row dataset. Indeed, in my experience with smaller row counts the .csv approach is faster, as dask has to take time to set up parallel jobs at the start. However, as the size of the data increases, one would expect the gap between the pandas+csv and dask+parquet approaches to start to diverge more, and the efficiency gains become more important.

Even beyond the time differences, I’m especially partial to dask+parquet and even dask+csv due to the difference in how much code is required at read-in. The fact that dask can read in an entire directory of data easily, in one line of code, is a hugely important feature for me.
