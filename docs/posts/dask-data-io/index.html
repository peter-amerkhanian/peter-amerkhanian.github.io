The following post covers some of my evolving thoughts on efficient data retrieval, data storage, and I/O tasks in Python. Accurately describing the cutting edge of the efficient Python landscape is a bit of a moving target, so the goal here is less to describe that cutting edge than to just give an overview of what I’m doing right now to balance efficiency and simplicity in my Python data science code.

I motivate this post with an analysis exercise where I retrieve and store historical data on Bay Area Rapid Transit (BART) ridership at the station/hour level. BART kindly makes such ridership information publicly available on their open data portal.

My end goal is to examine and justify (via benchmarking) the following workflows:

  1. Threaded downloads – use concurrent to distribute batch downloads across threads.
    Effect: ~20% decrease in download time.
  2. Column-oriented data storage – Use the apache parquet format to efficiently store data locally.
    Effect: ~50% reduction in data storage size.
  3. Threaded/Parallel Data Read-in – Use dask for efficient read-in of the data.

Our download workflow will be based on a for-loop wth the following logic built in:

  • Download each file from the list of links:
    • make sure that the ingest doesn’t re-download files that I already have downloaded locally.
    • UNLESS, it the data is from the current year, in which case it should be re-downloaded to reflect any updates since the last download.

These conditions should ensure that we don’t waste time and bandwidth re-downloading files that we already have, and that we always have the most up-to-date data for the current year.

def download_file(url:str, filename:str) -> str:
    response = requests.get(url, timeout=30)
    if response.status_code == 200:
        with open(filename, "wb") as file:
            file.write(response.content)
        return f"Downloaded: {filename}"
    else:
        return f"Failed ({response.status_code}): {url}"

def download_bart_file(url: str) -> str:
    current_year: str = str(datetime.today().year)
    filename: str = os.path.join("data", os.path.basename(url))
    file_year: str = re.search(r"\d{4}", url)
    current_year_data: bool = file_year and file_year[0] == current_year
    if os.path.exists(filename) and not current_year_data:
        return f"Skipped (already exists): {filename}"
    elif os.path.exists(filename) and current_year_data:
        download_file(url, filename)
    else:
        download_file(url, filename)

Threading

When downloading a series of files via a for-loop, thread-based parallelism can speed up the process (more on why in a moment). Thread-based parallelism, or, threading, involves breaking a process into smaller units (threads) and running them concurrently on shared memory to complete the larger process faster. To implement a a threaded workflow, I’ll use python’s ThreadPoolExecutor, a high-level library that makes it easy to distribute for-loop iterations across threads. I’ll also use tqdm to track the download progress.1

max_workers = 10 
with ThreadPoolExecutor(max_workers=max_workers) as executor:
    futures = [executor.submit(download_bart_file, url) for url in file_urls]
    for f in tqdm(as_completed(futures), total=len(futures)):
        f.result()

Is threading worth it?

Given the extra effort, it’s worth asking if setting up a multi-threaded download workflow is worth it. The following code chunk runs a simple comparison of a sequential for-loop approach vs threaded downloads of the BART data:

# Set up the output directory and parameters
OUTDIR = "data_bench"
MAX_WORKERS = 10  # number of parallel threads
os.makedirs(OUTDIR, exist_ok=True)

def make_filename(url: str) -> str:
    """Generate a local filename for each URL."""
    name = os.path.basename(url)
    if not name:
        name = "download"
    return os.path.join(OUTDIR, name)

def download_one(url: str) -> str:
    """Download a single file and save it to OUTDIR."""
    filename = make_filename(url)
    try:
        response = requests.get(url, timeout=30)
        if response.status_code == 200:
            with open(filename, "wb") as file:
                file.write(response.content)
            return f"Downloaded: {filename}"
        else:
            return f"Failed ({response.status_code}): {url}"
    except Exception as e:
        return f"Error downloading {url}: {e}"

# Benchmark functions
def reset_outdir():
    """Wipe and recreate the output folder."""
    if os.path.exists(OUTDIR):
        shutil.rmtree(OUTDIR)
    os.makedirs(OUTDIR)

def run_sequential(urls):
    """Download files one by one."""
    reset_outdir()
    start = time.perf_counter()
    for url in urls:
        download_one(url)
    end = time.perf_counter()
    return end - start

def run_parallel(urls):
    """Download files in parallel using ThreadPoolExecutor."""
    reset_outdir()
    start = time.perf_counter()
    with ThreadPoolExecutor(max_workers=MAX_WORKERS) as executor:
        futures = [executor.submit(download_one, url) for url in urls]
        for f in as_completed(futures):
            f.result()
    end = time.perf_counter()
    return end - start

def trial_download_one(url: str, download_dir: str) -> None:
    filename = make_trial_filename(url, download_dir)
    r = requests.get(url, timeout=30)
    if r.status_code == 200:
        with open(filename, "wb") as f:
            f.write(r.content)
    else:
        raise RuntimeError(f"HTTP {r.status_code} for {url}")

def reset_dir(path: str):
    if os.path.exists(path):
        shutil.rmtree(path)
    os.makedirs(path)

def run_sequential(urls, download_dir) -> float:
    reset_dir(download_dir)
    t0 = time.perf_counter()
    for u in urls:
        trial_download_one(u, download_dir)
    return time.perf_counter() - t0

    t0 = time.perf_counter()

        for f in as_completed(futs):
            f.result()
    return time.perf_counter() - t0

Why does threading help?

Our positive threading effect is likely due to the fact that the download process is I/O bound – much of the execution time is spent waiting for downloads to be completed. Threading allows us to run I/O bound processes like downloads concurrently. When one thread is waiting for a download to complete, another thread can be actively downloading a different file.

Threads are particularly useful when tasks are I/O bound, such as file operations or making network requests, where much of the time is spent waiting for external resources. – Python docs for threading

If the process were CPU bound – where most of the execution time is spent on computation – we would see less of or else no threading premium, and if we wanted to speed that up we would need to explore true parallel processing (e.g. via multiprocessing) rather than threading. That gets much more complicated, as CPU-bound processes often aren’t as obviously divisible as I/O for-loops, and achieving true parallelism in Python requires bypassing the “Global Interpreter Lock” – a topic for another day.

Column-Oriented Data Storage

Great, now we have the data downloaded locally. Now, how should we store it? I’d certainly like to minimize the amount of disk space it takes up, and also make sure that I can read it into memory efficiently when I want to analyze it. I’m going to advocate for storing the data in parquet format, a column-oriented data storage format developed by Apache. When we convert to parquet we’ll get two core benefits:

  1. The files will be smaller.
  2. The files can be read into memory much faster.

But don’t take my word for it – lets benchmark parquet against the original .csv.gz files and a .csv conversion.

Is parquet worth it?

Now that we have the files saved, we can conduct a quick check on how storage size differs. I’ll write a helper function to calculate the total bytes of all the files in a given directory.

def get_local_bytes(directory: str) -> int:
    contents: list[str] = os.listdir(directory)
    contents_w_path: list[str] = [os.path.join(directory, f) for f in contents]
    files_not_folders: list[str] = [f for f in contents_w_path
                         if os.path.isfile(f)]
    return sum(os.path.getsize(f) for f in files_not_folders)

Note that the very large discrepancy between .csv and .csv.gz is simple – the .csv.gz files are compressed (using gzip), whereas the .csv files are not. However, the parquet files are still significantly smaller than even the compressed .csv.gz files. This is where the parquet magic is – by storing data in a column-oriented format, it is able to compress the data more efficiently than simple gzip compression.

Why did parquet help?

Parquet files are smaller than .csv.gz (and certainly smaller than uncompressed .csv) because they store data by column instead of by row — this is called column-oriented storage. That means all the values from one column (like all the BART station names) are stored together, making patterns easier to spot and compress. Both parquet and .csv.gz are compressed formats, but a .csv.gz file, which is row-oriented, mixes everything together as text — numbers, commas, and words in each row — so it can’t shrink the data as efficiently. In short, Parquet saves space by organizing data by column and compressing repeated patterns.

Here I’ll use dask to read in about 52 million rows of data in well under 30 seconds, in a single line of code.

Note that the code for reading in the data, dd.read_parquet('data/parquet_data') executes lazily – it’s not reading data into memory yet! # Benchmark pandas vs Dask vs DuckDB on Parquet, CSV, and CSV.gz
# ------------------------------------------------------------
# pip install pandas pyarrow duckdb dask[complete] tqdm

import gc, time, statistics, glob
import pandas as pd
import duckdb
import dask.dataframe as dd

# --------------------- config ---------------------
PARQUET_GLOB = "data/parquet/*.parquet"
CSV_GLOB     = "data/csv/*.csv"
CSVGZ_GLOB   = "data/csv_gz/*.csv.gz"

GROUP_COL = "your_group_col"   # e.g., "station_id"
AGG_COL   = "your_value_col"   # e.g., "rides"
AGG_OP    = "sum"

REPEATS = 5
DUCKDB_THREADS = 8
# --------------------------------------------------

timer = time.perf_counter

def files_for(glob_pat):
    files = sorted(glob.glob(glob_pat))
    if not files:
        raise FileNotFoundError(f"No files for pattern: {glob_pat}")
    return files

# Collect results here
records = []

def run_and_record(fn, label, filetype, stage, repeats=REPEATS):
    """Run fn N times, record each duration to global records list."""
    for i in range(repeats):
        gc.collect()
        t0 = timer()
        _ = fn()
        dt = timer() - t0
        records.append({
            "engine": label.split()[0],
            "operation": label,
            "filetype": filetype,
            "stage": stage,
            "run": i + 1,
            "seconds": dt
        })
    print(f"{label} [{filetype}, {stage}] done.")

# ---------- Define workloads ----------
def read_pandas_parquet(): return pd.read_parquet(PARQUET_GLOB, engine="pyarrow")
def read_pandas_csv(glob_pat): return pd.concat((pd.read_csv(f) for f in files_for(glob_pat)), ignore_index=True)
def groupby_pandas_parquet(): df = read_pandas_parquet(); return df.groupby(GROUP_COL)[AGG_COL].agg(AGG_OP)
def groupby_pandas_csv(glob_pat): df = read_pandas_csv(glob_pat); return df.groupby(GROUP_COL)[AGG_COL].agg(AGG_OP)

def read_dask_parquet(): return dd.read_parquet(PARQUET_GLOB, engine="pyarrow").compute()
def read_dask_csv(glob_pat): return dd.read_csv(glob_pat, assume_missing=True).compute()
def groupby_dask_parquet(): ddf = dd.read_parquet(PARQUET_GLOB, engine="pyarrow"); return ddf.groupby(GROUP_COL)[AGG_COL].agg(AGG_OP).compute()
def groupby_dask_csv(glob_pat): ddf = dd.read_csv(glob_pat, assume_missing=True); return ddf.groupby(GROUP_COL)[AGG_COL].agg(AGG_OP).compute()

con = duckdb.connect()
con.execute(f"PRAGMA threads={DUCKDB_THREADS}")
def read_duckdb_parquet(): return con.sql(f"SELECT * FROM read_parquet('{PARQUET_GLOB}')").df()
def read_duckdb_csv(glob_pat): return con.sql(f"SELECT * FROM read_csv_auto('{glob_pat}')").df()
def groupby_duckdb_parquet(): return con.sql(f"SELECT {GROUP_COL}, {AGG_OP}({AGG_COL}) FROM read_parquet('{PARQUET_GLOB}') GROUP BY 1").df()
def groupby_duckdb_csv(glob_pat): return con.sql(f"SELECT {GROUP_COL}, {AGG_OP}({AGG_COL}) FROM read_csv_auto('{glob_pat}') GROUP BY 1").df()

# ---------- Run all ----------
def main():
    _ = files_for(PARQUET_GLOB); _ = files_for(CSV_GLOB); _ = files_for(CSVGZ_GLOB)

    print("=== READ → pandas ===")
    run_and_record(read_pandas_parquet, "pandas", "parquet", "read")
    run_and_record(lambda: read_pandas_csv(CSV_GLOB), "pandas", "csv", "read")
    run_and_record(lambda: read_pandas_csv(CSVGZ_GLOB), "pandas", "csv.gz", "read")

    run_and_record(read_dask_parquet, "dask", "parquet", "read")
    run_and_record(lambda: read_dask_csv(CSV_GLOB), "dask", "csv", "read")
    run_and_record(lambda: read_dask_csv(CSVGZ_GLOB), "dask", "csv.gz", "read")

    run_and_record(read_duckdb_parquet, "duckdb", "parquet", "read")
    run_and_record(lambda: read_duckdb_csv(CSV_GLOB), "duckdb", "csv", "read")
    run_and_record(lambda: read_duckdb_csv(CSVGZ_GLOB), "duckdb", "csv.gz", "read")

    print("\n=== READ → groupby → pandas ===")
    run_and_record(groupby_pandas_parquet, "pandas", "parquet", "groupby")
    run_and_record(lambda: groupby_pandas_csv(CSV_GLOB), "pandas", "csv", "groupby")
    run_and_record(lambda: groupby_pandas_csv(CSVGZ_GLOB), "pandas", "csv.gz", "groupby")

    run_and_record(groupby_dask_parquet, "dask", "parquet", "groupby")
    run_and_record(lambda: groupby_dask_csv(CSV_GLOB), "dask", "csv", "groupby")
    run_and_record(lambda: groupby_dask_csv(CSVGZ_GLOB), "dask", "csv.gz", "groupby")

    run_and_record(groupby_duckdb_parquet, "duckdb", "parquet", "groupby")
    run_and_record(lambda: groupby_duckdb_csv(CSV_GLOB), "duckdb", "csv", "groupby")
    run_and_record(lambda: groupby_duckdb_csv(CSVGZ_GLOB), "duckdb", "csv.gz", "groupby")

    # -------- Save results --------
    df_results = pd.DataFrame(records)
    df_results["seconds"] = df_results["seconds"].astype(float)

    print("\n=== Summary ===")
    summary = df_results.groupby(["engine", "filetype", "stage"])["seconds"].describe(percentiles=[0.5])
    print(summary)

    # Save for later analysis
    df_results.to_csv("benchmark_results.csv", index=False)
    print("\nSaved detailed results to benchmark_results.csv")

if __name__ == "__main__":

Some reflection

These differences in seconds aren’t so massive in the context of this 52 million row dataset. Indeed, in my experience with smaller row counts, just using .csv.gz/.csv with pandas is faster, as dask has to take time to set up parallel jobs at the start, and a parquet conversion has a fixed time cost up front. However, as the size of the data increases, one would expect the gap between the dask + parquet approach to start to diverge from the others more, and the efficiency gains become more important.

Even beyond the time differences, I’m especially partial to dask+parquet due to the difference in how much code is required at read-in. I also love the fact that dask can read in an entire directory of data easily in one line of code:

df = dd.read_parquet('data/parquet_data').compute()
